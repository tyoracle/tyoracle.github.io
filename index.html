10G RAC 常见VIP超时处理

发表于 2017-07-18 | 分类于 oracle,troubleshooting |

版本声明：本文为tyoracle.com原创，如需转发，无需经我同意，但必须要注明来自于tyoracle.com

前言

今天在客户那，遇到一套AIX环境下的10.2.0.4 RAC因为网卡闪断，而致VIP报错切换到另一个节点，使本机的service,listener等全部offline的问题，所以在此收集一下，各种情况下的VIP超时的处理方法，供日后问题查找与分析。

分析

因为客户没法把日志发给我，只能从网上找到类似的日志

crsd 日志报错

2012-04-16 00:57:43.437: [ CRSAPP][11321]32CheckResource error for ora.node02.vip error code = 1
2012-04-16 00:57:43.437: [ CRSRES][11321]32In stateChanged, ora.node02.vip target is ONLINE

如果问题会经常出现，建议设置以下的trace进行问题根因定位

VIP tracing is set by using the following commands:

#crsctl debug log res "ora.node01.vip:5"
#crsctl debug log res "ora.node02.vip:5"

取消trace可以用

#crsctl debug log res "ora.node01.vip:0"
#crsctl debug log res "ora.node02.vip:0"

原因定位

通过分析trace,发现原来是在AIX下的racgvip里存在这样一段脚本

# timeout of ping in number of loops (1 sec)
PING_TIMEOUT=" -c 1 -w 1"

其意思是

-S hostname/IP addr
Uses the IP address as the source address in outgoing ping packets.

-c Count
Specifies the number of echo requests, as indicated by the Count
variable, to be sent (and received).

-w timeout
This option works only with the -c option. It causes ping to wait
for a maximum of 'timeout' seconds for a reply (after sending the
last packet).

所以，上述的命令可以理解成，如果从公网IP 去ping 网关，返回时间超1秒即认为会报错

ping -S 192.168.220.36 -c 1 -w 1 192.168.220.33
==>192.168.220.36 是公网IP, 192.168.220.33 是网关.

正因为racgvip里有这个网络检查机制，在上述的情境中，因为AIX的网卡闪断，导致了RAC里的日志报超时2.434S，所以超出了racgvip里的检查阀值，必然会导致VIP超时，从而出现offline unexpectedly并漂移到另外的节点。

改善方法：

-1. 修改racgvip，将超时时间设大一点，至5秒（根据实际情况确定）



    1. Stop all node applications.
% srvctl stop nodeapps -n <hostname>

2. Backup then Modify the racgvip script .

Change:
# timeout of ping in number of loops (1 sec)
PING_TIMEOUT=" -c 1 -w 1"

To:
# timeout of ping in number of loops (5 sec)
PING_TIMEOUT=" -c 1 -w 5"

3. Start the node applications and other necessary resources.
% srvctl start nodeapps -n <hostname>



2、使用load balance模式的双网卡绑定  


只有使用load balance模式的双网卡绑定才可以，如果使用主备模式的绑定是不可以的，在实际情况遇到过，因为主备模式是会在主网卡失效后，重试3次，每次间隔1秒才会切换到备网卡，所以这个时候也会遇到超时3秒的情况。但使用load balance模式需要交换机支持。


另一个AIX下的VIP案例
适用于10.2.0.1–11.1的AIX下的RAC版本


srvctl start nodeapps -n racnode2
racnode2:ora.racnode2.vip:Interface eth0 checked failed (host=racnode2.us.oracle.com)
racnode2:ora.racnode2.vip:Invalid parameters, or failed to bring up VIP (host=racnode2.us.oracle.com)
CRS-1006: No more members to consider
CRS-0215: Could not start resource 'ora.racnode2.vip'

这是因为公网网卡用的是 LHEA (IBM Logical Host Ethernet Adapter):


# /usr/bin/entstat -d en0
-------------------------------------------------------------
ETHERNET STATISTICS (en0) :
Device Type: Host Ethernet Adapter (l-hea)
...

Or, the adapter type is null in entstat -d output:


# /usr/bin/entstat -d en3 
-------------------------------------------------------------
ETHERNET STATISTICS (en3) :
Device Type:
...

这是因为LHEA使用的entstat模式不同，其命中了Bug 8725020 - VIP WONT RUN (LHEA) ADAPTER 5.3 TL9


修复的方法可以修改racgvip的脚本


1. Backup the racgvip script
2. Edit this line in the script using vi:

$ENTSTAT -d $_IF | $GREP -iEq '.*lan.*state.*:.*operational.*|.*link.*status.*:.*up.*|.*port.*operational.*state.*:.*up.*'

and replace it with this:

$ENTSTAT -d $_IF | $GREP -iEq '.*lan.*state.*:.*operational.*|.*link.*status.*:.*up.*|.*port.*operational.*state.*:.*up.*|.*driver.*flags.*:.*up.*'

重启整个RAC生效。

Change the mode of executable which uses shared library from "delay binding" to "immediate binding" using this command:

$ chatr -B immediate <executable> 

So we need to do "chatr -B immediate" on following executables from both CRS Home and RDBMS Home:
crs_relocate.bin 
crs_start.bin 
crs_stop.bin 
crsd.bin 
evmd.bin 
racgons.bin 
racgeut 
racgevtf 
racgmain 

When finished start Clusterware and repeat the steps on the next node

Whenever you apply a patch or patchset to your installation repeat the above workaround

前言
今天我们来介绍一下另一个数据库方面的知识，MYSQL在今天已成为互联网的主流，在互联网行业里占比最高，也是数据库领域里一个热门话题。今天我们来介绍一下，在5.6的版本以上，如何使用基于时间点的恢复更加快速。


如果对mysql有些了解的都知道。MYSQL需要实现时间点的恢复，可以在物理备份后，加上binlog回放来实现恢复到某个时间点。所以全备+binlog就是一个标准的备份集，就像ORACLE的 RMAN+ARCHIVELOG一样。


MySQL binlog日志记录了MySQL数据库从启用日志以来所有对当前数据库的变更。binlog日志属于二进制文件，我们可以从binlog提取出来生成可阅读的SQL语句来重建当前数据库以及根据需要实现时点恢复或不完全恢复。


在传统的时间点恢复中，假设每天一个全备，备份是在凌晨12点备的，但因为某些误操作，在23：50分删除了一个大表，或者删除了某些重要的记录，这个时候，你就要用昨晚12点的备份加一整天的binlog进行恢复。如果按照常规的方式，那就非常缓慢，因为其利用的是mysqlbinlog命令，其速度的快慢，依赖于数据库在一天里有多少数据需要处理。
并且mysqlbinlog是一个单线程，是顺序恢复，所以其非常慢。


在这里，我想介绍一下并行恢复的思路。


在5.6以上的版本，我们可以转想一下思路，利用slave SQL_thread的并行恢复方式来加快应用二进制日志


测试
假设我们有一台配置了binary logs模式的单服务器，为了保证测试的真实性，我们使用sysbench以OLTP的方式运行着8个线程，对8个表进行操作，以模拟业务的正常运行。


此时，我们创建一个测试表:

mysql> create table myusers (id int auto_increment primary key, name varchar(20));

mysql> insert into myusers values (0,’lefred’),(0,’kennito’),(0,’dim0’);

mysql> insert into myusers values (0,’flyer’),(0,’luis’),(0,’nunno’);
Query OK, 3 rows affected (0.13 sec)
Records: 3 Duplicates: 0 Warnings: 0


mysql> select * from myusers;

| id | name    |
+—-+———+

| 2 | kennito  |
| 3 | dim0     |
| 4 | flyer    |
| 5 | luis     |
| 6 | nunno    |

6 rows in set (0.05 sec)




现在对数据库进行一个在线备份

[root@mysql1 mysql]# /opt/mysql/meb-4.1/bin/mysqlbackup –host=127.0.0.1 \
                  –backup-dir=/tmp/backup –user=root backup-and-apply-log


这个时候 sysbench仍然继续运行着，我们继续往这个表里插入数据


mysql> insert into myusers values (0,’alfranio’);
Query OK, 1 row affected (0.33 sec)


mysql> insert into myusers values (0,’vitor’);
Query OK, 1 row affected (0.09 sec)


这个时候，这个表共有8条记录，这时，误操作来了。。。。


某个操作人员，执行了一条delete语句，并且忘了加where条件，惨剧发生了，但是业务还不知道，继续往里面继续插入数据。



mysql> delete from myusers;
Query OK, 8 rows affected (0.23 sec)
and we don’t realize it directly, so we continue…


mysql> insert into myusers values (0,’pedro’);
Query OK, 1 row affected (0.19 sec)


mysql> insert into myusers values (0,’thiago’);
Query OK, 1 row affected (0.16 sec)

当维护人员检查记录时才发现，怎么这个表只有两条记录呢？原来的记录去那了？

通过上面的命令，我们可以看到delete操作发生在GTID：7766037d-4d1e-11e7-8a51-08002718d305:25076


我们想要做的，就是避免这个事务的执行，跳过这个事务。这时，我们需要停止MYSQL的服务，把二进制日志复制到另外一个目录，并且需要将最近的全备进行恢复。



[root@mysql1 ~]# systemctl stop mysqld
[root@mysql1 mysql]# mkdir /tmp/binlogs/


[root@mysql1 mysql]# cp mysql1-bin.* /tmp/binlogs/


cd /var/lib/mysql
[root@mysql1 mysql]# rm -rf 
[root@mysql1 mysql]# /opt/mysql/meb-4.1/bin/mysqlbackup –backup-dir=/tmp/backup copy-back
[root@mysql1 mysql]# chown -R mysql. 


 


这时，我们还需要修改一下my.cnf，增加以下选项：
It’s time to add some required settings in my.cnf:



replicate-same-server-id=1
skip-slave-start
 


修改完成后，就进行启动数据库，这时会恢复到备份时的状态，即表还是存在着6条记录。



[root@mysql1 mysql]# systemctl start mysqld
…
mysql> select * from sbtest.myusers;
+—-+———+
| id | name    |
+—-+———+
|  1 | lefred  |
|  2 | kennito |
|  3 | dim0    |
|  4 | flyer   |
|  5 | luis    |
|  6 | nunno   |
+—-+———+
6 rows in set (0.10 sec)



从上面看到，表的确是存在与备份时一致的记录，但是，我们需要进行基于时间点的恢复，还需要进行后面的操作恢复。


执行跳过误操作的GTID

mysql> SET @@GLOBAL.GTID_PURGED=’7766037d-4d1e-11e7-8a51-08002718d305:25076’;


现在，我们需要将二进制日志变成relay 日志，所以我们需要先将早前复制的二进制日志进行改名。


[root@mysql1 mysql]# for i in $(ls /tmp/binlogs/.0)
do
  ext=$(echo $i | cut -d’.’ -f2);
 cp $i mysql1-relay-bin.$ext;
done


然后将所以改名后的relay日志，记录到 mysql1-relay-bin.index 这个文件。


[root@mysql1 mysql]# ls ./mysql1-relay-bin.0 >mysql1-relay-bin.index
更改权限
[root@mysql1 mysql]# chown mysql. relay*
数据库开始应用这些relay 日志。
mysql> CHANGE MASTER TO RELAY_LOG_FILE=’mysql1-relay-bin.000001’,
       RELAY_LOG_POS=1, MASTER_HOST=’dummy’;
Query OK, 0 rows affected (4.98 sec)


这时我们就能利用到relay的并行复制进行恢复了。


如果您没有将工作负载分布在多个数据库中，从5.7起，在启动“SQL_THREAD”之前，最好使用与默认值不同的从属并行类型：


mysql> SET GLOBAL SLAVE_PARALLEL_TYPE=’LOGICAL_CLOCK’;
mysql> SET GLOBAL SLAVE_PARALLEL_WORKERS=8;


现在可以使用新的中继日志启动复制：


mysql> START SLAVE SQL_THREAD;


使用以下的语句监控并行应用的状态


mysql> select * from performance_schema.replication_applier_status_by_worker\G


在进行完基于时间点的恢复后，可以看到，我们跳过了误操作，关于这个表的所有相关都恢复出来了。


mysql> select * from sbtest.myusers;

| id | name     |
+—-+———-+
|  1 | lefred   |
|  2 | kennito  |
|  3 | dim0     |
|  4 | flyer    |
|  5 | luis     |
|  6 | nunno    |
|  7 | alfranio |
|  8 | vitor    |
|  9 | pedro    |
| 10 | thiago   |
+—-+———-+
 


如果你只想恢复到产生误操作的时间点，而后面的操作不再恢复，也是可以的。


恢复的步骤与上述类似，将binlog日志改名为relay log，但这时我们不需要进行 GTID_PURGED跳过误操作的事务。我们只需要执行以下的操作，指定until到某一个GTID



mysql> CHANGE MASTER TO RELAY_LOG_FILE=’mysql1-relay-bin.000001’,
       RELAY_LOG_POS=1, MASTER_HOST=’dummy’;
mysql> set global slave_parallel_type=’LOGICAL_CLOCK’;
mysql> SET GLOBAL SLAVE_PARALLEL_WORKERS=8;
mysql> START SLAVE SQL_THREAD UNTIL
       SQL_BEFORE_GTIDS = ‘7766037d-4d1e-11e7-8a51-08002718d305:25076’;


这一次，我们将复制到指定的GTID然后停止SQL_THREAD。


在这两种情况下，请勿忘记执行PTR后，重置所有从属信息：


mysq> RESET SLAVE ALL;
 


结论
当然这不是PTR的标准方法。 通常，人们使用mysqlbinlog并通过MySQL的客户端重播它进行恢复。 但这 失也是一个比较特别的方法，在某些情况下可能会节省大量的时间。

前言
最近在看一些关于parallel的相关ppt，计划写一定这方面的东西，这些都是比较基础的东西，但是需要概念清晰。并行执行是通过将任务分解成更小的子任务来加速操作的常用方法。


那我们为什么需要使用并行呢？


假设，我们需要计算停在一条马路上汽车有多少辆，我们可以一个人从头到尾，慢慢数，直至清点完，你也可以请你的一个朋友帮忙，让他从另一边数过来，直至你们两个在路中间相遇。假设你和你朋友的清算速度一样，这样就相当于节省了原来的一半时间，这就是并行的好处。


在ORACLE里有什么操作可以使用并行：


    

  • Access methods. For example, table scans, index full scans, and partitioned index range scans.
    • 

  • Join methods. For example, nested loop, sort merge, hash, and star transformation.
    • 

  • DDL statements. CREATE TABLE AS SELECT, CREATE INDEX, REBUILD INDEX, REBUILD INDEX PARTITION, and MOVE SPLIT COALESCE PARTITION
    • 

  • DML statements. For example, INSERT AS SELECT, updates, deletes, and MERGE operations.
    • 

  • 其他杂项的操作. For example, GROUP BY, NOT IN, SELECT DISTINCT, UNION, UNION ALL, CUBE, and ROLLUP, as well as aggregate and table functions.
    • 



通常并行大多数是在数据仓库环境中使用，因为数仓都是大数据，大的批处理。有时在OLTP环境中，使用适当的并行，如批处理，并行DDL，也能获得不错的效果。

什么时候需要推进SCN？
在启动时出现下面的错误时，需要推进SCN




* 2017-04-01 12:46:03.768
ORA-00704: bootstrap process failure
ORA-00704: bootstrap process failure
ORA-00604: error occurred at recursive SQL level 1
ORA-01555: snapshot too old: rollback segment number 33 with name    “_SYSSMU33_2839196578$” too small
ORA-00704: bootstrap process failure
ORA-00704: bootstrap process failure
ORA-00604: error occurred at recursive SQL level 1
ORA-01555: snapshot too old: rollback segment number 33 with name “_SYSSMU33_2839196578$” too small




这时可以在pfile.ora里尝试加入以下这些参数：




.undo_management=MANUAL      
._allow_resetlogs_corruption=true
._no_recovery_through_resetlogs=true   
._corrupted_rollback_segments=’_SYSSMU33_2839196578$’
*.event=”10513 trace name context forever, level 2” 




首先，报ORA-01555肯定 需要加大SCN的，原来我们试过加10000，后来加到100000都不行，直接加到1千万。   


另外的经典错误：
ERROR:
ORA-600 [2662] [a] [c] [d] [e]


也是需要推进SCN解决  


在这里，总结每一种不同系统推进的方法，方便以后直接查阅


SCN小知识
scn是由scn base及scn wrap构成的，当scn base达到一定程度，scn wrap则会递增，一般情况下scn wrap是0，不会变化
scn base及scn wrap也是数据块中的数据结构，可见scn base是4个字节，而scn wrap是2个字节
  也就是说scn base要用4个字节用完，scn wrap就会递增
  依理推理，4个字节为 power(2,32),也就是达到这个数据时，scn base就会归0，scn wrap递增1


计算公式：
(SCN_WRP * 4294967296) + SCN_BAS should give us the SCN in the number format


举例

SCN: 14441583156
15:12:29 sys@DCNPOS5G> select dbms_flashback.get_system_change_number a from dual;
  14441583156
Elapsed: 00:00:00.00


select to_char(14441583156,’xxxxxxxxxxx’) from dual;
  35cc91234
计算：
SYS@PROD2>select 14441583156/4294967296 from dual;
            3.3624431


SYS@PROD2>select to_char(14441583156-3*4294967296,’xxxxxxxxx’) from dual;
  5cc91234
其SCN_WRP 为3, SCN_BAS为 5cc91234




其在文件头中的位置如下（如果需要单独修改某一个数据文件，SCN只需要修改这两个地方)



BBED> p kcvfh
   struct kcvfhckp, 36 bytes                @484
      struct kcvcpscn, 8 bytes              @484
         ub4 kscnbas                        @484      0x028f5c58  –scn base
         ub2 kscnwrp                        @488      0x0000      –scn wrap




推进SCN的方法
1、使用oredebug poke (本文主要讲这一种）  


2、alter session set events ‘10015 trace name adjust_scn level 1’; （10015 本身仅仅是一个事件， 它在当smon做transaction recovery时被触发，实际造成scn 增长的是 adjust_scn 这个事件；是指 在触发10015 事件时 做adjust_scn 这个action 调整scn ）


3、利用alter system set “_MINIMUM_GIGA_SCN”=6 scope=spfile;
_MINIMUM_GIGA_SCN推进SCN，1代表2的30次方，值为1073741824，如你希望将SCN推进至
6442451243左右，即将上述参数设为6


上述方法各有优缺点，但在11.2.0.3以上的版本，使用poke的方法更方便快捷


oradebug poke说明
oradebug poke可以推进SCN，分为在数据库OPEN及MOUNT皆可以
oradebug setmypid
oradebug DUMPvar SGA kcsgscn     （获取当前内存中的scn)
但是一般我们只会在mount的时候使用,在mount时，oradebug DUMPvar SGA kcsgscn得到的值会全是0，这是因为数据库没有打开。


查看某一个地址的当前值可以使用oradebug peek 0x3497DE1C 300


另外要注意，使用oradebug poke进行SCN递增，没有高低字节转换，直接to_char出来的值，直接加0x就可以,AIX的加法，与LINUX不同，32位与64位的增加方法也不同

总结
推进SCN，是万不得已的操作，推进的过程往往需要结合加入其他隐含参数，重建控制文件的动作，所以具体问题具体分析。但是对于SCN的相关问题，并没有想像中复杂，需要修改的地方就那么几个地方，细心加耐心，保证方向正确，肯定能把库拉起来。

每天还需要面对工作上各种各样的琐碎小事，IT人的上班时间绝不只是上班的那个八小时，更多的是八小时以外的工作与学习。


现代大都市，上下班的时间越来越长了，当一天的工作完成，从客户处拖着疲惫的身心回到家的时候，还有一堆的文档要写，一堆的资料要翻阅，一堆的邮件等着你去回复，还要再学点什么东西的情况下，基本上没有时间了。所以你看到这个网站上的文章更新得很慢。有时在问自己，我为什么要写这个呢？

                  你看清自己的目标了吗?
                
              
            
          

        

        

          
            
              
                
              
              
                发表于
              
              
                2017-03-17

1
很多人在每年开始的时候，都会给自己定下一个今年的职业目标，如今年取的绩效是多少？要完成什么任务？然后就冲着这个目标埋头工作了。一年下来，任务达到了欢喜，没有达到就失落.—-但是有没有问问自己，为什么要完成这个目标呢？为什么是这个任务呢?期间自己到底有没有增值？我们可能完全没有深入思考过，或者我们一直是低水平的勤奋。


2
你每天这么努力地做时间管理，各种挤出累得要死的下班后2小时做思考学习，但是最大段的8小时时间，却交给了一个根本不知你人生方向，也许也不知道你优劣势的管理者的分配，是不是挺滑稽的？


最大的时间管理，其实是方向上的管理—-我们要学会自己做定位，结合组织要求来工作，这样才是上班，不是让班上你：-）
什么是战略：绘制地图（从广州到珠峰）—洞察力，选择路径—决策力，当下一步—执行力，抬头看路–适应力


3
愿景就是梦想，使命就是职业，价值观就是道德。


愿景就是一幅描绘了美好未来的图景，你要想想，因为你的存在，10年，20年后世界是什么样子的，画面感越强，越能激发憧憬。当然，改变世界，是高级愿景，但可以不改变世界，而是成就自己。不管是改变世界还是成就自己。只要是真的心向往之，总能点燃激情。就不要怕愿景太大，万一实现了呢。


使命就是我为什么而存在，到底在做什么，你的人生目标是什么？在实现改变世界的愿景的路上，扮演什么角色。


所以我们要做的是，给自己一个美好的愿景，清晰自己的使命，这样才能有目标不断努力奋斗。
