ORA-600 [kcbzwfcro_2]

发表于 2017-03-01 | 分类于 oracle,bug |   |   次阅读

版本声明：本文为tyoracle.com原创，如需转发，无需经我同意，但必须要注明来自于tyoracle.com

背景

新年开始，一切都变得繁忙，所以一直没有时间去写博文，离今年的目标还远着呢。最近两天一个客户在升级到了11.2.0.4最新的PSU后，在每天晚上跑批时，都会报大量的ORA-00600: internal error code, arguments: [kcbzwfcro_2]报错，客户原以为升级至最新的PSU就不会遇到BUG，还是遇到了坑，虽然没有让数据库DOWN机，但每晚半夜的报警声也让他无法安睡。所以让我帮忙分析一下，看能不能解决。

从call stack里看，报错是出现在事务层，在做一个undo recovery时，被中止后报出这个错语。对于bug，我们没有什么好的想法时，只能求助于metalink，但是在metalink上，在当前版本11.2.0.4与kcbzwfcro_2相关的，只有一个bug相关，见下图

这个bug是指在when querying a dropped table会出现ORA-600 [kcbzwfcro_2],其原理是因为Force-cr-override ,Force-cr-override 是从11.2.0.3引进来的一个特性，是指当一个buffer要求一个强制的一致性读时，在这个时候，看到一个current状态的buffer
在群集的另一个节点时，我们从另一个节点复制这个current状态的buffer并覆盖这个强制的一致性读的buffer。如果这个buffer块重新使用时，
缓存熔合将会使在整个群集中的这个buffer block包括这个强制性一致读的buffer变得无效。但这个情况下有一个很少的时间间隔。
所以在这个间隔里，会容易出现问题。如果我们发现潜在的陈旧缓冲区是一个强制重写缓冲区，等待最大5分钟，以确保缓存融合有时间来获得force-cr重写缓冲区，
但如果时间上不允许，这个时候就会出现buffer 的不一致，从而报出ORA-00600[kcbzwfcro_2]的错误。


Description

A query on a dropped segment could signal an ORA-600 [kcbzwfcro_2]

Rediscovery Notes
  ORA-600[kcbzwfcro_2] in a trace file or the alert.log

Workaround
  None

但是很不幸，这个bug的描述与客户环境并不一致，客户并没有删除表，call stack与这个bug的描述也不相关。从客户的trace里看，是基于事务的current buffer block不一致，需要Force-cr-override ，从而引起了ORA-00600的报错。在客户的trace里，看到的确一直在等待Force-cr-override

这个时候真的没有什么思绪了，因为与客户版本吻合的只有一个bug，但这个bug并现有场景并不一致。只能继续在搜索，通过ktdabt+ktdAbortMultinodeTransaction 搜索到一个相近的

bug:22593521
ORA-600: [kcbzwfcro_2], [83592], [1], [32768], [0] was raised when
doing “rollback force” on distributed transaction.

这个bug的bh dump是相似的，也是dump undo block，但是其callstack是不相似的，所以也不是完全匹配。

对于这种情况，既然在官方没有找到匹配的，一般只能开一个SR，提交相关的trace给官方，然后漫长的等待确认是BUG，这个时候他们还不能给你 patch，必须要等上一段时间才有patch，这对于生产系统来说，这是非常严重的事，需要想办法避免这个bug才可以。

要避免这个bug，只能复盘，看看这个Bug的原理，上文已经提到了，这个报错是因为Force-cr-override的特性引起的。 这个错误可能是由于cache_cr_override缓冲区上的高速缓存融合问题而出现，或由于空间层问题（像块被不同的重用）对象或缓冲区已损坏）所致。

既然是知道了引起问题的特性，那我们只能想办法把这个force-cr-override关闭，是不是可以绕过这个bug?关闭这个,高速缓存融合对current buffer block的管理将会回到11.2.0.3以前的常规方式，对数据库的性能影响应该是比较少的。所以与客户沟通后，决定关闭这个特性

alter system set “_gc_override_force_cr”=false scope=both ;

无需重启，参数即可生效。

结论

参数设置后，几天都没有再出现相关的ORA-00600错误，当然这个还得长期观察 。

当你遇到ORA-00600或07445的时候，绝大部分的情况是遇到了BUG，你只需到metalink上去确认，是不是这个是不是BUG，什么情况下是确认这个BUG与你的吻合呢？首先，版本，平台，肯定要一致吧？产生的原因也类似吧？如是不是因为rollback引起的或commit引起的， 还需要验证call stack是不是一致。如果上述都吻合，那你的库命中这个bug的概率达90%以上，如果这个bug有提供WORKAROUND方法，那么恭喜你，你可以应用WORKAROUND推荐的方法，从而解决这个报错。

但是如果很不幸，你在metalink上找不到跟你的报错相符合的bug，那你只能开个SR，慢慢等待确认。要不你就通读整个trace，找出产生这个bug的原理，想办法绕过它。

上述两点，基本上是解决BUG的思路，希望能对你的DBA生涯有所帮助。
